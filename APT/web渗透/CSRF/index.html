CSRF

流程

所谓网站的交互，任何功能都是发送数据包实现的

  1. 想要篡改A网站的某用户信息
  2. 搭建了一个B网站，完后JS留了修改A网站个人资料的数据包
  3. 骗取用户在登录A网站的情况下来访问B网站
  4. 无意间访问了B网站的某个页面，触发了修改A网站个人资料的数据包
  5. 浏览器发现是访问了A网站的会把这个用户在A网站已登录成功的Cookie添加到数据包中
  6. A网站一看Cookie正确，就以为是本人发起的这个请求，就会修改成功

成因

  1. CSRF漏洞造成的原因就是网站的Cookie在浏览器关闭时不会过期
  2. Cookie一般的过期时间是一次会话，一次会话就是关闭浏览器
  3. Cookie存在于浏览器中，但是你不可能只访问一个网站，所以浏览器中的Cookie不止一个
  4. 当访问网站的时候，浏览器会判断你访问的什么站，发送相应的数据包，其中就包含该网站的Cookie

防御手段

验证码

CSRF核心是 跨域 + 伪造

跨域 : 通过在服务端对HTTP请求头Referer字段进行检查

伪造核心 : 请求中的所有参数都可以预测

伪造 : 客户端页面添加伪随机数（验证码、一次性Token）

验证码是最好的防御方法，修改密码，需要验证码，任何操作都使用验证码验证可以完全杜绝CSRF

验证referer

由于校验 referer 一般是通过正则表达式来进行匹配，开发人员可能存在疏忽，导致了被绕过的可能（正常情况下验证 referer 是不能绕过的）

token字段

开发为了安全，设置一个，每个请求表单中都存在一个字段，这个字段就是token（可以利用xss获取token）

绕过手段

绕过referer

  1. 判断referer是否存在某关键词（即匹配referer值中有无此网站域名）

有时候，他的referer验证只是匹配referer中有无该域名

xxx.com网站存在一个csrf，通过burp编写poc，讲代码保存到本地html文件中，而这个html的文件名就可以是referer验证的域名，抓包获取后，referer可能是 http://127.0.0.1/xxx.com.html 这就可以绕过referer验证了

  1. 判断referer是否有域名（即判断referer开头是否以xxx.com以及该子域，并没有验证根域名，所以可以构造xxx.com.123.com来绕过）

空 referer

测试时就是给数据包中的 referer 置空即可，有些网站可能没有考虑到 referer 为空的情况或者业务需要 referer 为空的情况从而导致了绕过

如果要实际利用，那么可以使用以下的一些方法

  1. form 表单页面加上如下内容：
<meta name="referrer" content="never" >
  1. a 标签的一个属性 ref
<a href="xxx" ref="noreferrer">TEST</a>
  1. 利用其他的协议，比如data:、file:等
<iframe src="data:text/html;base64,PGZvcm0gbWV0aG9kPXBvc3QgYWN0aW9uPWh0dHA6Ly9hLmIuY29tL2Q+PGlucHV0IHR5cGU9dGV4dCBuYW1lPSdpZCcgdmFsdWU9JzEyMycvPjwvZm9ybT48c2NyaXB0PmRvY3VtZW50LmZvcm1zWzBdLnN1Ym1pdCgpOzwvc2NyaXB0Pg==">

如果目标是 http 的站点，那么将 poc 放到 https 的站点上，从 https 的站点跳转到 http 的站点，也是不带有 referer 的

关键词绕过

  1. 查看是否必须存在指定的关键词
  2. 如必须存在abc.com，那么我们只需要使用xxx.com/poc.html?abc.com即可绕过
  3. ?也可以换成其他的一些符号，只要不影响 html 解析就行，比如#

指定域绕过

相比上一个更难，比如 referer 必须在域abc.com下，这种情况也有一些方法：

  1. 使用@，比如abc.com@xxx.com/poc.html
  2. 看看子域名有不有发布文章的功能的地方，比如存在一个子域名forum.abc.com可以发布文章，那么把 poc 发布到这个域名下诱导其他人访问也是可以的
  3. 如果是 GET 型 CSRF，思路和第二条也差不多，只是要简单很多，找个能从网站内访问 POC 的点即可，比如网站头像处设置成 CSRF 的 POC
  4. 找一个任意 URL 跳转的漏洞来结合
  5. 结合 XSS

总结就两种

  1. 第一种就是空 referer
  2. 第二种就是和任意 URL 跳转一样的绕法

JSONP

当这个接口没有验证Referer头的时候，就存在JSONP劫持漏洞，即在任何域下都能窃取到传输的数据，当接口返回的是一些敏感数据时（如csrf token、个人用户信息等）危害是很大的

Flash CSRF

flash跨域权限管理文件设置为允许所有主机/域名跨域对本站进行读写数据

xxx.com/crossdomain.xml

<cros-domain-policy>
    <allow-access-from domain="*"/> // 错误配置
</cros-domain-policy>
