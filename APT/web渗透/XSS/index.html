# XSS 分类

# 反射型

提交的数据只对本次访问产生了影响

  1. 攻击者将 payload 放置在 url 链接中（这是针对是 GET 型反射 XSS）
  2. 用户点击该恶意链接
  3. XSS 服务将 XSS 代码（JavaScript 代码）以及视图返回给客户端
  4. 客户端解析视图以及 XSS 代码（JavaScript 代码），并将执行结果发送到 XSS 平台
  5. 攻击者访问 XSS 平台，读取用户的敏感信息（Cookie）

# 存储型

提交的数据存入了数据库，别人访问这个页面的时候就会自动获取数据（不一定是数据库，也有可能是存在于日志中）

  1. 攻击者向 XSS 服务插入 XSS 代码
  2. XSS 服务会将其结果存储到数据库中
  3. 用户正常访问 XSS 服务
  4. XSS 服务将数据库的数据以及视图返回给前端，前端渲染视图并加载数据，其中数据里包含恶意 XSS 代码（JavaScript 代码）
  5. 客户端渲染视图，加载 XSS 代码，并向攻击者的 XSS 服务发送敏感信息
  6. 攻击者读取用户的敏感信息

# DOM 型

基于文档对象模型 Document Object Model 的一种漏洞，DOM 是一个与平台，编程无关的接口，允许脚本动态的访问

  1. 攻击者将 payload 放置在 url 链接中（这是针对是 GET 型反射 XSS）
  2. 用户点击恶意链接，并打开浏览器
  3. 此时浏览器客户端并不会发起 http 请求到 XSS 服务，而是在浏览器客户端执行 XSS（JavaScript 代码）
  4. 此时将 XSS 代码执行结果发送给攻击者的恶意服务
  5. 攻击者访问自己的 XSS 平台并读取用户的敏感信息

# Cookie、Session

# session

  1. session 从字面上讲就是会话，类似和某个人在交流，但是如何确定这个交流的对象就是想要交流的人，对方肯定具有，某种特征表明
  2. 为了做这种区分，服务器需要给每个客户端分配不同的身份标识，然后客户端每次向服务器发请求的时候都会携带这个身份表示，服务器自然就知道这个请求来自谁
  3. 至于客户端怎么保存这个身份表示，有很多种方式，但对于浏览器客户端，默认采用 cookie 的方式，服务器使用 session 把用户的信息临时保存到服务器上，用户离开后 session 会销毁
  4. 但是这种存储方式相比 cookie 更安全，但是如果 XSS 服务器做了负载均衡，那么下个操作请求到了另一台服务器时 session 会丢失

# cookie

  1. cookie 是一个非常具体的东西，就是浏览器能永久存储的一种数据，仅仅是浏览器实现的一种数据存储功能，cookie 由服务器生成后发送给浏览器
  2. 浏览器把 cookie 以 kv 格式保存到某个目录下的文本文件内

# XSS 总结

分类

反射型：提交的数据只对本次访问产生了影响
存储型：提交的数据会被存储到数据库中
DOM 型：基于 DOM 文档对象进行攻击

cookie 和 session 区别

session 是存储在服务器中，更加安全，主要是给每个客户端分配不同的身份标识
cookie 是浏览器能永久存储的一种数据，它是由服务器生成后发送给浏览器，浏览器以 kv 格式保存到某个目录下的文本内

绕 WAF

先打入测试语句，然后慢慢分析可能拦截哪些或者放通哪些 标签替换、特殊符号（#）、单引号替换尖括号、DOM 型 XSS 中 script => scri+t、fuzz 测试

xss 防护

输入输出编码、加 WAF、开启 http-only
