# URLDNS 链

当目标服务器存在一个反序列化的点时，那么可以通过来 URLDNS 链，验证服务器存在 SSRF

# 漏洞发现

# URL

由于 URL类 ​继承了 Serializable ​接口，那么就可以进行序列化和反序列化操作

​

但是，发现它重写的 readObject ​中没有一些其他函数的调用，此时发现这个序列化无法行得通，但是它这个类还重写了其他方法​

​

我们更进这个 hashCode() ​函数

​

发现它调用了 getHostAddress(u) ​方法，从函数名字可以看出是一个发起 DNS 请求的函数，此时触发函数就找到了

现在考虑的问题就是如何调用这个 hashCode() ​方法

# HashMap

找到常用的 HashMap ​查看其有无重写该方法

​

发现先通过调用 hash() ​方法，然后在调用它的 hashCode() ​方法，那么哪里有调用了 hashMap ​的 hash ​方法

​

发现它的 readObject ​调用了 hash ​方法，最终通过序列化 hashMap ​就能成功执行 hash ​方法

# 构建链路

​

但是当执行完序列化后发现在 burp 已经接受到了请求，这里好像出了点问题

​

发现 hashMap.put () 执行时就已经触发了 hash ​函数导致在序列化时无法生效

那么我们想要实现：在 put 时，让它无法发起请求，在反序列化时又可以发起请求，这时有需要利用它的一个 if 条件判断

​

在 URL 类中存在一个变量 hashCode ​只有它的值不为 - 1 时才不会执行 hashCode() ​，在使用反射的时候就可以将它的值进行修改，然后 put 完成后，再将它的值修改为 - 1

​

当执行反序列化后，看它是否会触发

成功执行 DNS 请求

# 总结

  1. 通过 HashMap 的 readObject () 方法执行 hash () 方法
  2. hash () 方法会接收一个参数，就是 HashMap 的 key
  3. 在创建 HashMap 的时候传入的键类型为 URL 对象
  4. 而 hash () 内部又会执行 key.HashCode ()
  5. 如果这个 key 是 URL 类型，则会调用 URL 类中的 hashCode 方法
  6. hashCode 中存在可以发起请求的方法，导致 SSRF 执行

其中会存在一个问题就是，在 new HahsCode 后，需要往其中添加数据，而添加数据的函数是 put，这个 put 也会触发 hash 函数，此时需要通过反射修改 URL 类中的 hashCode 变量的值，让它不为 - 1（默认是 - 1）在反序列化之前将它的值改回 - 1 即可成功执行 SSRF
