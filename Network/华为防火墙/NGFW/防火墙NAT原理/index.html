网络 NAT 转换原理

NAT 技术通过对 IP 报头中的原地址或目的地址进行转换，可以使大量的私网 IP 地址通过共享少量的公网 IP 地址来访问公网

产生背景

  1. IPV4 地址日渐枯竭
  2. IPV6 技术不能立即大面积替换
  3. 各种延长 IPV4 寿命的技术不断出现，NAT 就是其中之一
    优点
  4. 实现 IP 地址复用，节约宝贵的地址资源
  5. 地址转换过程对用户透明
  6. 对内网用户提供隐私保护
  7. 可实现对内部服务器的负载均衡
    缺点
  8. 网络监控难度增大
  9. 限制某些具体应用

NAT 分类

源 NAT（SNAT）

  1. Easy IP(出接口地址方式)
  2. NAPT(地址池方式)
    一般用于内网访问外网使用（上网）数据发出时转换数据的源（IP、端口）数据返回时转换数据的目的（IP、端口）

目的 NAT（DNAT）

  1. NAT Server（静态映射）
    一般用于外网访问内网使用（开放服务）数据发出时转换数据目的（IP、端口）数据返回时转换数据的源（IP、端口）

NAPT

需要一个地址池，地址池中至少有一个地址（这个地址池也可以只存放一个接口公网地址）

将不同的内部地址映射到同一公有地址的不同端口号上，实现多对一地址转换

转换过程

即 172.16.1.100:2051 => 100.100.100.100:80 变成了 202.96.137.89:2049 => 100.100.100.100:80

NAT Server

  • 外网主动访问内部服务器
    内部服务器（Nat Server）功能是使用一个公网地址来代表内部服务器对外地址，而在防火墙上，专门为内部的服务器配置一个对外的公网地址来代表私网地址，对于外网用户来说，防火墙上配置的外网地址就是服务器的地址

Server-Map 表

当配置 NAT Server 时，设备会自动生成 Server-Map 表项，用于存放 global 地址与 inside 地址的映射关系

[FW]nat server protocol tcp global 202.96.137.88 80 inside 192.168.1.100 80
[FW]nat server protocol icmp global 202.96.137.88 inside 192.168.1.100

当配置 NAT Server 后，会自动生成 Server-Map 表项，其中存在正向和反向 NAT 策略

从表中得知一个是正向 NAT Server，表示任何地址访问公网地址（202.96.137.88:80）都会跳转到 192.168.1.100
另一个是反向 NAT Server Reverse，表示我们内网服务器（192.168.1.100）可以访问任何地址时，源 IP 转为公网 IP（202.96.137.88）

这样对内网服务器来说是不安全的（服务器可能会主动上网），反向的这条记录相当于是一个源 NAT 策略，可以在创建 NAT Server 时添加参数 no-reverse 来指定只添加正向（加入 no-reverse 参数后就可以配置多条指向同一内网地址的 NAT Server）

[FW]nat server 0 protocol tcp global 202.96.137.88 www inside 192.168.1.100 www no-reverse
[FW]nat server 1 protocol icmp global 202.96.137.88 inside 192.168.1.100 no-reverse

多出口 NAT Server

当存在多条链路可以访问内网，就需要配置多条 NAT Server

但是如果 Server-Map 中存在指向同一出口的反向 NAT Server 的话，会阻止创建第二条 NAT Server

202.96.137.88:80 => 192.168.1.100:80
113.100.1.99:80 => 192.168.1.100:80

此时有两条链路，第一条创建完后会存在反向 NAT 192.168.1.100 => 202.96.137.88，假设第二条也可以创建，即也会存在反向 NAT 192.168.1.100 => 113.100.1.99

当外网访问公网接口IP时会找到对应的映射服务器，但是服务器在回数据包时发现 NAT 表中存在两条NAT，此时就会不知道该给220.96.137.88转发还是113.100.1.99转发，所以只能禁止配置第二条 NAT Server

这时需要在新的 NAT Server 上添加 no-reverse 参数

添加 no-reverse

配置NAT Server时需要在后面添加no-reverse参数即可禁止生成反向 NAT (NAT 配置好后也得配置安全策略放行流量即 DX/LT => dmz 的流量)

// 第一条链路
[FW]nat server protocol tcp global 202.96.137.88 www inside 192.168.1.100 www
[FW]nat server protocol icmp global 202.96.137.88 inside 192.168.1.100

// 添加第二条链路
[FW]nat server protocol tcp global 113.100.1.99 www inside 192.168.1.100 www no-reverse 
[FW]nat server protocol icmp global 113.100.1.99 inside 192.168.1.100 no-reverse

这里设置的是服务器可以发送 icmp 和 http 但是得通过 202.96.137 进行转发，抓包查看数据

配置不同区域

创建NAT Server时可以使用绑定区域的方式来进行生成对应的反向NAT

[FW]nat server zone DX protocol tcp global 202.96.137.88 www inside 192.168.1.100 www
[FW]nat server zone DX protocol icmp global 202.96.137.88 inside 192.168.1.100
[FW]nat server zone LT protocol tcp global 113.100.1.99 www inside 192.168.1.100 www
[FW]nat server zone LT protocol icmp global 113.100.1.99 inside 192.168.1.100

抓包查看流量，都显示一个接口（这是都设置了正反 NAT Server）

双向 NAT

  • 同时改变源IP和目的IP

域间双向NAT

为了简化配置服务器至公网的路由，可在NAT Server基础上，增加源NAT 配置（比如内网存在大量服务器，网关可能经常发生变动，服务器不需要配置网关）即通过利用二层通信来实现访问服务器（NAT Server转换源IP，Easy-IP转化目的IP）

转换流程

域内双向NAT

用户通过公网地址访问内网服务器
防火墙将用户的请求报文的目的地址转换成服务器的内网IP，源地址转换成用户对外公布的IP地址
防火墙将服务器的回应报文原地址转换成对外公布的地址，目的地址转换成用户的内网IP地址

转换过程

  1. 只配置NAT Server

因此光转换目的IP不行，还需要将其源IP进行转换，这样链路就能正常访问

总结

  1. NAPT

内网用户通过地址池转换源IP来访问公网，缺点就是地址池地址必须是固定的（接口地址固定）

  1. Easy-IP

通过公网接口IP来访问公网
